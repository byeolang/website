1. 빌드 벽 없는 개발

Title
빌드 벽 없는 개발

Description
빌드를 기다리느라 흐름을 끊지 않도록.
코딩, 반영, 디버깅이 하나의 빠른 루프로 이어지는 개발을 지향합니다.

첨부될 차트 / 그래프 설명
일반적인 개발 흐름과 byeol의 개발 흐름을 나란히 비교하는 흐름 다이어그램.
왼쪽은 코드 작성 → 빌드 → 대기 → 실행 → 디버깅, 오른쪽은 코드 작성 → 바로 확인 → 계속 수정 형태로 구성한다.
막대그래프보다, 단계 수와 대기 구간 차이가 한눈에 보이는 파이프라인 비교 형식이 적합하다.

2. 적은 문법, 빠른 이해

Title
적은 문법, 빠른 이해

Description
문법을 공부하는 시간보다 아이디어를 코드로 옮기는 시간이 더 길어지도록.
byeol은 적은 문법 요소로 같은 일을 표현하는 언어를 지향합니다.

첨부될 차트 / 그래프 설명
두 요소를 함께 배치한다.
첫째, 키워드 수 비교: guide 기준 byeol 키워드 수는 18개다. 
둘째, 짧은 코드 비교: 동일한 기능을 하는 C# 코드와 byeol 코드를 4~5줄 정도로 나란히 보여준다.
차트는 숫자만 강조하지 말고, “같은 로직을 더 적은 문법 요소로 표현한다”는 점이 드러나도록 코드 비교와 함께 구성한다.

비교 예시

C#

class Player { int hp = 9; public bool IsDanger() => hp < 10; } Console.WriteLine(new Player().IsDanger()); 

byeol

def player hp := 9 isDanger() bool: hp < 10 print(player.isDanger()) 

3. 스크립트처럼 가볍게, 오류는 먼저

Title
스크립트처럼 가볍게, 오류는 먼저

Description
가볍게 시작할 수 있는 스크립트 언어이지만,
실수는 실행 전에 최대한 먼저 잡아내도록.
빠른 실험과 정적 검사를 함께 노립니다.

첨부될 차트 / 그래프 설명
전통적인 그래프보다 코드 예시 + 오류 표시 형식이 적합하다.
짧은 코드 조각에 대해, 실행 전에 탐지된 타입 불일치나 잘못된 호출 같은 오류가 표시되는 형태로 구성한다.
핵심은 “실행 후 실패”가 아니라 “실행 전 감지”를 직관적으로 보여주는 것이다.

4. AI 이후에도, 계속 만들고 싶은 언어

Title
AI 이후에도, 계속 만들고 싶은 언어

Description
AI가 더 많은 코드를 쓰는 시대에도,
사람이 직접 만지고, 고치고, 만들고 싶어지는 언어.
byeol은 생산성과 감성을 함께 목표로 합니다.

첨부될 차트 / 그래프 설명
차트나 그래프는 넣지 않는다.
이 카드는 기능 증명이 아니라 프로젝트의 철학과 방향성을 전달하는 카드로 사용한다.
숫자보다 메시지 자체가 중심이 되어야 한다.
